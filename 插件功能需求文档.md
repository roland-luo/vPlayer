## 问题描述

vPlayer 已完成基础播放链路（M1），具备播放/暂停、进度控制、音量、播放列表、错误处理等能力。下一步需要引入可扩展的插件系统，使 vPlayer 能从"能用的播放器"进化为"可扩展的播放平台"。

插件系统是后续所有特色功能（截图、字幕下载、学习辅助等）的基础设施。

## 需求范围

核心交付物：一个支持 Rust 动态库插件的运行时框架，包含加载、生命周期管理、事件总线、权限控制、前端集成、错误隔离。

覆盖的插件类型：
- **视频处理类** — 截图、录屏、画质分析等
- **字幕/音频辅助类** — 自动字幕下载、翻译、配音等
- **播放管理类** — 播放历史、收藏、统计等
- **AI/学习辅助类** — 字幕总结、章节标记、学习笔记等

## 前提条件（已确认）

1. **架构路线**：先 Rust 动态库插件，后续扩展到 JS/脚本
2. **前端集成**：框架提供统一按钮 + 弹窗容器，插件不提供自定义组件
3. **权限模型**：只限制文件系统和网络访问
4. **打包格式**：动态库（`.dylib`/`.so`/`.dll`）+ `manifest.json`
5. **生命周期**：全热插拔（启动加载 + 运行时启/禁 + 热安装/卸载）
6. **覆盖类型**：视频处理、字幕/音频、播放管理、AI/学习辅助
7. **首发内置插件**：截图、字幕下载、学习辅助
8. **分发渠道**：先本地目录，后续在线商店
9. **配置管理**：插件独立配置文件 + 统一管理界面
10. **错误隔离**：`catch_unwind` 隔离，单插件崩溃不影响其他

## 方案：分阶段交付（推荐）

### Phase 1 — 插件系统 MVP

**目标：** 基于现有 `bus.rs` + `registry.rs`，实现最小可工作的插件加载和通信系统。

#### 新增模块

| 模块 | 文件 | 说明 |
|------|------|------|
| 动态加载器 | `src/plugin/loader.rs` | 基于 `libloading` 加载动态库，调用插件生命周期函数 |
| Manifest 解析 | (可合入 loader.rs) | 解析插件目录下的 `manifest.json` |
| 权限检查器 | `src/plugin/sandbox.rs` | 对文件写/网络操作做声明式检查（最小版本） |
| 插件管理前端 | `frontend/src/views/PluginManager.vue` | 插件列表、启用/禁用、配置入口 |
| 插件弹窗容器 | `frontend/src/components/PluginPopup.vue` | 统一弹窗容器，承载插件 UI 内容 |

#### 插件 Manifest 格式

```json
{
  "name": "screenshot",
  "version": "1.0.0",
  "description": "快速截图保存为 PNG",
  "author": "vPlayer Team",
  "permissions": ["file_write"],
  "ui": {
    "button": {
      "label": "截图",
      "icon": "camera",
      "position": "control_bar"
    },
    "popup": {
      "width": 400,
      "height": 300
    }
  },
  "api_version": 1
}
```

#### 插件 Rust trait

```rust
pub trait Plugin: Send + Sync {
    fn name(&self) -> &str;
    fn on_load(&mut self, ctx: &PluginContext) -> Result<(), PluginError>;
    fn on_unload(&mut self) -> Result<(), PluginError>;
    fn on_event(&mut self, event: &PluginEvent) -> Result<(), PluginError>;
}
```

#### Phase 1 数据流

```
[~/.config/vPlayer/plugins/] → loader.rs 扫描目录
    → 读取 manifest.json
    → libloading 加载 .dylib/.so/.dll
    → 调用 on_load() 初始化
    → 注册到 registry.rs
    → bus.rs 开始分发事件
    → 前端通过 IPC 获取插件列表并展示
```

#### 前端插件弹出流程

```
用户点击插件按钮
    → 前端 emit plugin:open_popup {name}
    → 后端调用插件 on_open()（扩展 trait）
    → 后端返回插件 UI 内容（可通过 IPC 传入数据）
    → 前端 PluginPopup 容器展示内容
    → 用户关闭弹窗 → 前端 emit plugin:close_popup
```

#### Effort 估算

- 后端（loader + manifest + sandbox + bus 扩展）：3-5 天 / CC 2-3h
- 前端（管理面板 + 弹窗容器 + IPC）：2-3 天 / CC 1-2h
- Completeness：6/10（最小可行，跳过热插拔和完整权限 UI）

### Phase 2 — 热插拔 + 权限 + 管理界面

**目标：** 支持运行时动态加载/卸载插件，完整的权限管理和前端界面。

#### 新增能力

- **目录监听**：使用 `notify` crate 监控 `plugins/` 目录变化，检测新增/删除后自动热加载/卸载
- **运行时启/禁**：用户可随时启用/禁用已安装插件，无需重启播放器
- **权限确认**：首次安装时显示权限清单，用户确认后才激活
- **热插拔流程**：
  1. 文件系统监听 detect 新 `.dylib` + `manifest.json`
  2. `loader.rs` 验证 manifest、加载动态库、调用 `on_load()`
  3. 若 `on_load()` 失败 → 自动回滚，不注册该插件
  4. 若成功 → `registry.rs` 注册，发送 `plugin:installed` 事件
  5. 文件删除同理：调用 `on_unload()` → 从 registry 移除
- **管理面板完善**：权限列表、配置编辑、日志查看
- **Effort：** M / Completeness：8/10

---

### Phase 3 — SDK + API 版本化 + 在线商店

**目标：** 建立完整的插件生态基础设施。

#### 新增能力

- **Plugin SDK**：独立的 Rust crate `vplayer-plugin-sdk`，提供插件开发所需宏和 trait
- **API 版本化**：manifest 中声明 `api_version`，播放器启动时做兼容性检查
- **在线商店发现**：插件列表浏览、一键安装
- **签名验证**：确保插件来源可信（开发模式可跳过但提示风险）
- **使用量统计**：可选上报，帮助开发者了解插件活跃度
- **Effort：** XL / Completeness：10/10

## 关键技术决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 动态库加载 | `libloading` crate | Rust 生态最成熟的动态加载方案，跨平台 |
| 热插拔文件监听 | `notify` crate | 跨平台文件系统监控，macOS/Linux/Windows |
| 序列化 | `serde` + `serde_json` | 项目已引入，manifest/config 统一复用 |
| 插件 trait 约束 | `Send + Sync` | 插件可能被多线程事件总线访问 |
| 前端弹窗方案 | Vue dynamic component + Teleport | 复用现有模式，无新依赖 |
| 配置格式 | TOML（每个插件独立文件） | 比 JSON 更人类可读，Rust 原生支持 |

## 错误处理策略

| 场景 | 处理方式 | 用户可见性 |
|------|----------|-----------|
| 动态库加载失败（格式错误/符号缺失） | 记录日志，跳过该插件 | 低（列表标灰 + 错误原因） |
| `on_load()` 崩溃 | `catch_unwind` 捕获，自动禁用 | 中（Toast 提示"插件加载失败"） |
| `on_event()` 崩溃 | `catch_unwind` 捕获，标记异常 | 中（Toast 提示 + 错误计数） |
| 权限越权（插件尝试未声明操作） | 拦截 + 记录日志 | 低（静默拦截） |
| 插件死循环/阻塞 | `invoke()` 设置 5s 超时 | 高（超时弹窗 + 自动标记 suspect） |
| manifest 解析失败 | 跳过该插件目录 | 低（日志记录 + 列表显示"manifest 无效"） |

## 前端集成

### 插件按钮注册

```
loader.rs 加载插件 → 读取 manifest.ui
    → 通过 IPC event "plugin:ui_register" 通知前端
    → 前端在 ControlBar 或菜单栏对应位置渲染按钮
    → 用户点击按钮:
        → 前端 emit "plugin:open_popup" {name}
        → 后端调用插件的 on_open() 获取数据
        → 前端 PluginPopup 容器渲染
        → 用户关闭 → 前端 emit "plugin:close_popup"
```

### 插件管理页面

- **列表视图** — 名称、版本、作者、状态（启用/禁用/已崩溃）
- **权限详情** — 声明了哪些权限（文件读、文件写、网络）
- **配置编辑** — 每个插件的独立配置文件，TOML 格式
- **日志查看** — 插件运行日志（错误、崩溃、越权尝试）
- **操作** — 启用/禁用、卸载、打开配置目录

### 状态同步事件（后端 → 前端）

| 事件 | 载荷 | 触发时机 |
|------|------|----------|
| `plugin:installed` | `{name, version}` | 插件安装完成 |
| `plugin:uninstalled` | `{name}` | 插件卸载完成 |
| `plugin:state_changed` | `{name, enabled}` | 启用/禁用切换 |
| `plugin:error` | `{name, code, message}` | 插件运行错误 |
| `plugin:ui_register` | `{name, button}` | 插件注册 UI 按钮 |

## 期望目录结构

```
~/.config/vPlayer/
├── plugins/                  # 插件安装目录
│   ├── screenshot/           # 每个插件一个子目录
│   │   ├── manifest.json
│   │   ├── libscreenshot.dylib   # macOS
│   │   ├── screenshot.conf       # 插件配置（TOML）
│   │   └── ...
│   └── subtitle-dl/
│       ├── manifest.json
│       └── libsubtitle_dl.dylib
├── plugin-logs/              # 插件运行日志
│   ├── screenshot.log
│   └── subtitle-dl.log
└── vplayer.conf              # 播放器主配置
```

## 我注意到你的思维方式

- 你对插件类型有清晰的优先级区分，没有"什么都要"的冲动扩张，这是 founder 式的聚焦思维
