# vPlayer 代码审查流程

> 适用范围：vPlayer 桌面端播放器（Tauri + Rust + libmpv）的所有代码变更。
> 审查重点：内存安全、IPC 边界、GPU/OpenGL 资源管理、插件隔离机制。

---

## 1. 审查前准备

### 1.1 提交规范检查

- [ ] PR 标题简明，格式：`[<领域>] <动词> <对象>`，例：`[renderer] 修复纹理分配失败时的内存泄漏`
- [ ] 关联 Issue 或设计文档（如有）
- [ ] 变更范围清晰，单次 PR 不超过 400 行核心逻辑（重构除外）
- [ ] `cargo fmt` 与 `cargo clippy` 无警告
- [ ] 前端代码通过 `eslint` / `prettier` 检查

### 1.2 环境与依赖

- [ ] 新增依赖已评估许可证与体积影响
- [ ] `Cargo.toml` / `package.json` 变更说明充分

---

## 2. 审查维度清单

### 2.1 Rust 后端（Tauri Command / libmpv 绑定）

| 维度 | 检查项 | 严重级别 |
|------|--------|----------|
| **内存安全** | `unsafe` 块是否有充分注释与边界检查；是否存在 use-after-free | 🔴 阻塞 |
| **错误处理** | `Result` 是否被正确处理，禁止裸 `.unwrap()` / `.expect()` 在生产路径 | 🔴 阻塞 |
| **资源管理** | OpenGL 纹理、FBO、mpv render context 是否确保释放（`Drop` 实现） | 🔴 阻塞 |
| **并发安全** | 跨线程传递的句柄是否实现 `Send + Sync`；锁的粒度与死锁风险 | 🟡 建议 |
| **IPC 边界** | Tauri Command 的输入是否校验（路径遍历、非法字符、超大载荷） | 🔴 阻塞 |
| **性能** | 是否存在渲染主线程阻塞（I/O、重分配、锁竞争） | 🟡 建议 |

### 2.2 前端（Web 视图层）

| 维度 | 检查项 | 严重级别 |
|------|--------|----------|
| **状态管理** | 播放器状态机是否清晰，避免与后端状态双源真理 | 🟡 建议 |
| **错误展示** | 是否对接后端 `video:error` 事件，提供用户可见的错误 Overlay | 🟡 建议 |
| **DOM 性能** | 高频更新（进度条、音量）是否节流 / 使用 RAF | 🟢 提示 |
| **无障碍** | 关键控件是否有 `aria-label`，键盘可操作 | 🟢 提示 |

### 2.3 插件系统（plugin / bus）

| 维度 | 检查项 | 严重级别 |
|------|--------|----------|
| **隔离边界** | 插件崩溃是否被 `try/catch`（或 Rust 等价物）捕获，不影响宿主 | 🔴 阻塞 |
| **权限模型** | 插件是否只获得最小必要能力（文件访问、网络、渲染） | 🔴 阻塞 |
| **生命周期** | 插件加载 / 卸载是否清理资源，避免泄漏 | 🟡 建议 |

### 2.4 测试覆盖

- [ ] 新增 Rust 逻辑包含单元测试（`cargo test`）
- [ ] 涉及 IPC 的变更包含集成测试（Tauri `invoke` 模拟）
- [ ] 渲染路径变更在本地通过手动验证（至少覆盖常规播放 + 全屏切换）
- [ ] 错误路径有测试（如纹理分配失败、`libmpv` 初始化失败）

## 3. 审查执行流程

```
开发者提交 PR
    │
    ▼
┌──────────────────────┐
│ CI 自动检查          │  → fmt / clippy / eslint / test
└──────────────────────┘
    │ 通过
    ▼
┌──────────────────────┐
│ 审查者分配           │  → 至少 1 名 Rust + 1 名前端（跨领域变更）
└──────────────────────┘
    │
    ▼
┌──────────────────────┐
│ 逐行审查             │  → 使用 GitHub Suggestion / Comment
│ + 维度清单核对       │
└──────────────────────┘
    │
    ├── 发现 🔴 阻塞问题 ──→ 开发者修复 → 重新审查
    │
    ├── 发现 🟡 建议问题 ──→ 开发者回复处理方案（修复 / 解释 / 转 Issue）
    │
    └── 通过 ─────────────→ 审查者 Approve → 合并
```

---

## 4. 特殊场景审查要点

### OpenGL / 渲染相关变更

- 是否在所有代码路径释放 GL 资源？（包括错误提前返回）
- `mpv_render_context_render()` 的返回值是否被检查？
- 窗口 resize / 全屏切换时纹理重新分配的策略是否合理？

### libmpv 集成变更

- `mpv_create()` / `mpv_initialize()` 失败时是否有回退逻辑？
- 是否向前端发送友好错误信息而非静默崩溃？
- 多线程调用 libmpv API 是否符合其线程安全模型？

### 插件系统变更

- 事件总线 `emit()` / `invoke()` 是否为每个插件包装了错误边界？
- 插件崩溃后是否自动禁用并记录日志？
- 是否向用户显示非侵入式通知？

---

## 5. 审查后跟踪

1. **合并后观察**：涉及渲染或启动路径的变更，合并后 24h 内观察 CI 与 canary 日志
2. **问题回溯**：若合并后发现回归，审查者参与复盘，更新审查清单
3. **清单迭代**：每季度回顾本清单，根据项目演进补充新检查项
