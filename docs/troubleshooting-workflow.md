# vPlayer 故障排查流程

> 适用范围：vPlayer 桌面端播放器（Tauri + Rust + libmpv）运行时故障的系统性排查。
> 核心原则：先隔离层级（OS / GPU / libmpv / 前端 / 插件），再逐层深入。

---

## 1. 信息收集

收到故障报告后，首先收集以下上下文：

| 信息项 | 获取方式 | 用途 |
|--------|----------|------|
| 应用版本 | 关于页面 / 日志头 | 判断是否为已知问题 |
| 操作系统与版本 | 日志 / 用户上报 | 平台特异性问题定位 |
| GPU 型号与驱动版本 | 启动日志 / `dxdiag` / `glxinfo` | OpenGL / 渲染相关故障 |
| 故障现象 | 用户描述 + 录屏 | 区分 UI 假死、黑屏、崩溃、无响应 |
| 复现步骤 | 用户操作序列 | 判断是否依赖特定文件 / 操作路径 |
| 日志文件 | `%APPDATA%/vPlayer/logs` 或 `~/.config/vPlayer/logs` | 崩溃栈、错误码、GL 上下文状态 |
| 插件列表 | 设置页面 / 日志 | 排除第三方插件干扰 |

---

## 2. 故障分类与初步定界

根据现象快速归类：

```
故障报告
    │
    ├── 启动阶段崩溃 / 无法打开窗口 ──→ 跳转 3.1 启动故障
    │
    ├── 播放时黑屏 / 花屏 / 无画面 ──→ 跳转 3.2 渲染故障
    │
    ├── 播放卡顿 / 音画不同步 ──→ 跳转 3.3 解码与性能故障
    │
    ├── UI 无响应 / 操作失灵 ──→ 跳转 3.4 前端与 IPC 故障
    │
    ├── 某个插件导致异常 ──→ 跳转 3.5 插件故障
    │
    └── 其他异常 ──→ 跳转 3.6 通用排查
```

---

## 3. 分层排查步骤

### 3.1 启动故障（libmpv / OpenGL 初始化失败）

**典型现象**：双击应用无窗口弹出、闪退、显示错误页面。

| 步骤 | 操作 | 判定 |
|------|------|------|
| 1 | 检查日志中 `mpv_create()` 返回值 | 非 0 表示 libmpv 核心未初始化 |
| 2 | 检查 OpenGL 上下文创建日志 | 失败常见于 GPU 驱动过旧、虚拟机、远程桌面 |
| 3 | 收集 `glxinfo` / `dxdiag` 中的 GL 版本 | 低于 3.0 需引导用户升级驱动 |
| 4 | 在兼容模式（软件渲染）下尝试启动 | 若正常则为 GPU 驱动 / 硬件问题 |
| 5 | 检查是否缺少运行时依赖（如 `libmpv.so` / DLL） | 打包或安装问题 |

**已知模式**：
- **GPU 驱动过旧**：日志中出现 `GLXBadFBConfig`、`WGLCreateContext` 失败 → 引导用户更新驱动
- **虚拟机 / 远程桌面**：无可用 GPU 加速 → 考虑回退到软解或软件渲染路径
- **TODOS 关联**：`libmpv 初始化失败的回退策略` 已实现友好错误页面，确认该页面是否正常触发

---

### 3.2 渲染故障（黑屏 / 花屏 / 纹理分配失败）

**典型现象**：UI 正常但视频区域黑屏、画面撕裂、分辨率切换后无画面。

| 步骤 | 操作 | 判定 |
|------|------|------|
| 1 | 检查日志中 `render_frame()` 返回值 | 非 0 表示渲染失败 |
| 2 | 检查纹理分配日志（宽高、格式、GPU 内存） | 失败常见于超大窗口或 GPU 内存不足 |
| 3 | 尝试调整窗口大小后重试 | 若恢复则与 resize 时的纹理重分配有关 |
| 4 | 切换视频输出驱动（`opengl` / `libmpv` 软件渲染） | 隔离 GPU 驱动兼容性 |
| 5 | 抓取一帧 GPU 调试信息（RenderDoc / Nsight） | 深层 GPU 状态异常 |

**已知模式**：
- **纹理分配失败**：日志中 `glTexImage2D` / `glFramebufferTexture2D` 错误 → 需向前端发送 `video:error` 事件，提示用户降低分辨率或关闭其他 GPU 应用
- **TODOS 关联**：`OpenGL 纹理分配失败的用户可见错误` 已实现错误 overlay，确认是否正常触发

---

### 3.3 解码与性能故障（卡顿 / 音画不同步 / 高 CPU）

**典型现象**：播放中频繁缓冲、画面掉帧、CPU/GPU 占用异常高。

| 步骤 | 操作 | 判定 |
|------|------|------|
| 1 | 查看日志中的 `buffering` 事件密度 | 高频 = 网络或解码瓶颈 |
| 2 | 检查 `mpv` 的 `hwdec` 状态 | 硬解未启用时 CPU 占用高 |
| 3 | 对比不同编码格式的表现（H.264 / H.265 / AV1） | 某些 GPU 对 AV1 硬解支持不佳 |
| 4 | 检查系统内存与 GPU 显存占用 | 资源不足导致回退到软解 |
| 5 | 使用 `perf` / `tracy` 抓取 Rust 后端热点 | 定位是否存在主线程阻塞 |

---

### 3.4 前端与 IPC 故障（UI 无响应 / 状态不一致）

**典型现象**：点击播放无反应、进度条不更新、状态显示与实际不符。

| 步骤 | 操作 | 判定 |
|------|------|------|
| 1 | 打开 DevTools（Tauri 调试模式）检查 Console | 前端 JS 错误 |
| 2 | 检查 Tauri IPC 通道 `invoke` / `listen` 是否正常 | 前端事件是否到达后端 |
| 3 | 验证后端是否向前端推送了状态更新（`tauri::Event`） | 后端事件丢失或格式错误 |
| 4 | 对比前端状态树与后端 `mpv` 实际状态 | 双源真理导致 UI 漂移 |
| 5 | 检查是否因高频 IPC 导致消息堆积 | 如逐帧进度更新未节流 |

---

### 3.5 插件故障

**典型现象**：安装某插件后功能异常、崩溃频率上升、特定操作触发错误。

| 步骤 | 操作 | 判定 |
|------|------|------|
| 1 | 安全模式下启动（禁用所有第三方插件） | 若正常则定位到插件 |
| 2 | 逐个启用插件，二分法定位 culprit | 缩小范围 |
| 3 | 检查插件崩溃日志与堆栈 | 插件内部错误 |
| 4 | 检查事件总线日志，确认隔离机制是否生效 | 插件崩溃不应拖垮宿主 |
| 5 | 验证插件权限是否越界 | 如非法文件访问、网络请求 |

**已知模式**：
- **插件拖垮宿主**：隔离机制未生效 → 检查 `src/plugin/bus.rs` 的 `emit()` / `invoke()` 错误边界
- **TODOS 关联**：`插件崩溃隔离` 已实现 try/catch 隔离，确认是否正常工作

---

### 3.6 通用排查

若以上分类均不适用：

1. **最小复现**：尝试用最简文件 / 最简操作复现，排除特定输入因素
2. **版本回退**：在上一稳定版本验证是否复现，确定是否为回归
3. **对比环境**：在另一台设备 / 虚拟机 / 干净系统上验证
4. **增加日志**：在可疑位置补充 `trace!` / `debug!` 日志，重新构建验证
5. **核心转储**：若进程崩溃，收集 coredump + 符号表，使用 `gdb` / `lldb` 分析栈帧

## 4. 修复与验证

| 阶段 | 动作 | 标准 |
|------|------|------|
| 定位根因 | 形成假设并通过实验验证 | 能在本地或测试环境稳定复现 |
| 修复实施 | 修改代码并补充测试 / 日志 | 不引入新的 🔴 阻塞级问题 |
| 回归验证 | 在故障环境 + 正常环境双重验证 | 原故障消失，原有功能不受影响 |
| 日志增强 | 若故障难以预判，增加诊断日志 | 下次同类故障可更快定位 |
| 文档更新 | 若为新发现模式，补充至本流程 | 形成组织知识 |

---

## 5. 上报与闭环

- **创建 Issue**：包含复现步骤、环境信息、日志片段、已尝试的排查手段
- **关联 TODOS**：若发现已知待办事项未覆盖的盲区，更新 `TODOS.md`
- **用户反馈**：修复后向报告者确认，收集满意度反馈

## 附录：快速检查命令

```bash
# 检查 libmpv 版本与支持的解码器
mpv --vd=help

# 检查 OpenGL 信息（Linux）
glxinfo | grep "OpenGL version"

# 检查 OpenGL 信息（Windows，通过 PowerShell）
# 使用 dxdiag 导出或第三方工具

# 抓取 vPlayer 运行时日志（Linux/macOS）
tail -f ~/.config/vPlayer/logs/latest.log

# 抓取 vPlayer 运行时日志（Windows）
Get-Content "$env:APPDATA\vPlayer\logs\latest.log" -Wait
```
